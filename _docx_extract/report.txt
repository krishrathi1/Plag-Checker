VeriCheck AI
Project Improvement Report
Side-by-Side Comparison with Turnitin — Roadmap to 10/10
Prepared for: Krish Rathi  |  April 2026
Document Classification: Development Guidance  |  Confidential
Executive Summary
VeriCheck AI is an impressively architected academic integrity platform. After a thorough code review of the entire codebase — server engine, client UI, detection pipeline, and infrastructure — this report benchmarks it directly against Turnitin's published and observed capabilities, identifies concrete gaps, and provides a prioritised roadmap to achieve a production-grade 10/10 score.
VeriCheck AI already demonstrates several capabilities that rival or exceed the MVP bar for an integrity checker. The multi-pass plagiarism engine (Winnowing + TF-IDF + Character Shingles), the AI detection ensemble, DOCX forensic metadata extraction, obfuscation normalisation, live web corpus expansion via OpenAlex/Crossref/Wikipedia, role-based access control, organisation management, and an async job queue with webhook notifications are all features that serious institutional products offer.
However, key structural gaps remain before this can be considered on par with Turnitin: the AI detection relies entirely on heuristic proxies without a real language model, the plagiarism corpus is ephemeral JSON (no persistent database), similarity thresholds are hand-tuned with no validation dataset, the frontend lacks highlighted sentence-level visualisation, there is no LTI / LMS integration, no similarity index across student submissions (cross-submission checking), and no test coverage. These gaps represent the difference between a strong prototype and a deployable institutional product.

3. Deep-Dive Analysis by Category
3.1  Plagiarism Detection Engine
What You Have: The engine in plagiarism.ts is genuinely well-engineered. It runs three passes — Winnowing fingerprinting (Schleimer/Aiken SIGMOD 2003), TF-IDF cosine similarity with smoothed IDF, and multi-resolution character shingles (k=5,6,9) — taking the maximum signal from all three. Corpus sentences are pre-indexed at scan start for performance.
What is Missing:
Semantic/embedding-based matching: TF-IDF is a bag-of-words model that cannot capture paraphrasing like 'canine' vs 'dog'. Turnitin uses BERT-style sentence embeddings. Adding a lightweight embedding model (e.g., BGE-small, MiniLM-L6, or the Sentence Transformers library) would detect paraphrase plagiarism that currently returns 0% similarity.
Cross-submission (inter-document) similarity: Turnitin's most powerful feature is its student submission index — comparing every new paper against all previously submitted papers in the institution. VeriCheck AI only checks against a manually-uploaded corpus, not against other students' submissions.
AST-based code similarity: The code engine currently does text-based matching. Turnitin and MOSS use Abstract Syntax Tree parsing, which detects variable renaming and structural reordering. Each language (Python, Java, C++) needs its own AST parser.
Calibration dataset: The scoring weights (0.35 weighted mean, 0.25 coverage, etc.) are hand-tuned. Without a labelled ground-truth dataset, it is impossible to know if thresholds are set correctly. Even a small 100-document labelled set would allow proper calibration.
3.2  AI Content Detection
What You Have: The AI detection engine in ai.ts is creative and shows genuine understanding of the problem. It computes 12 signals — lexical diversity, punctuation density, trigram repetition, connector phrase density, cross-perplexity ratio proxy, probability curvature proxy, volatility decay, derivative dispersion, and autocorrelation rhythm — and fuses them with a logistic meta-classifier. The isotonic calibration curves and strictness profiles are well thought-out.
Critical Limitation: None of these signals use an actual language model. They are surface-level statistical proxies that approximate what a real LLM-based detector would compute. The cross-perplexity ratio uses only two 24-word lookup tables (COMMON_WORD_PROBS_A and COMMON_WORD_PROBS_B), not actual GPT/LLaMA log-probabilities. This will produce significant false positives and false negatives. Turnitin uses a production-trained transformer classifier.
How to Reach 8+/10 on AI Detection:
Integrate Binoculars (open-source): A zero-shot LLM-based detector using two models — a scorer (Falcon-7B) and an observer — to compute the 'binocular score'. This is the closest open-source equivalent to Turnitin's AI detector and requires no training data.
Alternative: Use the OpenAI moderation API or GPTZero API for AI probability scoring, replacing the heuristic engine with real model output.
Alternative (lightweight): Use a pre-trained BERT classifier fine-tuned on HC3 or RAID datasets (available on HuggingFace) and call it via the Transformers.js or ONNX Runtime.
Keep the heuristic engine as a fast pre-filter: If the heuristic score is below 0.15, skip the expensive model call to reduce cost.
3.3  Corpus Coverage
What You Have: The live web corpus module queries OpenAlex (academic papers), Crossref (journal metadata), and Wikipedia. This is a smart architecture — free, real-time, and directly relevant. When ENABLE_LIVE_WEB_CORPUS=true, it fires up to 4 queries per submission, fetching up to 40 documents.
Gaps:
Academic coverage is abstract-only: OpenAlex and Crossref return only titles and abstracts, not full article text. Turnitin has licensed access to full-text journal content. For an open alternative, integrate CORE.ac.uk (145M open access papers with full text).
No general web search: Wikipedia covers encyclopaedic content but misses blogs, news sites, and online essays — common sources of student plagiarism. Integrating Brave Search API (free tier) or SerpAPI would dramatically improve recall.
Deduplication is title+URL hash only: Two sources with the same content but different URLs would both appear. A content-hash deduplication would reduce noise.
No persistent corpus indexing: Every submission rebuilds the TF-IDF index from scratch. With a growing corpus, this becomes slow. Pre-indexing the corpus and storing TF-IDF vectors would make scanning sub-second.
3.4  Frontend UI and Report Visualisation
What You Have: The React/Vite client includes a full report viewer with organisation stats, submission history, and sentence-level scores. The type definitions are clean and match the API contract.
Critical Gap — No Inline Sentence Highlighting: Turnitin's most distinctive UX feature is the split-pane view showing the original document with plagiarised sentences highlighted in colour, and a side panel showing the matched source. VeriCheck AI currently shows sentences as a flat list. This is the single highest-impact UI improvement.
Recommended UI improvements:
Add a document view panel with inline colour-coded sentence highlighting (red = high similarity, orange = moderate, yellow = low, green = AI-detected).
Show matched source snippets in a side panel when hovering over a highlighted sentence.
Add a similarity donut chart (Recharts is already in scope based on the React setup).
Add a timeline chart comparing multiple scan versions of the same document.
Make the PDF export match the on-screen report layout with colour-coded sections.
3.5  Persistence and Data Layer
Critical Issue: The entire database is a single JSON file (server/data/db.json), written and read synchronously. This is the most critical architectural gap. Problems with this approach: no concurrent write safety, a single corrupt JSON write can destroy all data, no querying capability (all filtering is done in memory), and the file will grow unboundedly as submissions accumulate.
Recommended Migration Path:
Immediate (dev/MVP): Migrate to SQLite using better-sqlite3. This is a single-file database that provides ACID guarantees, proper indexing, and supports the same serverless deployment model. The schema maps directly from the current types.ts interfaces.
Production (institutional): Migrate to PostgreSQL with Prisma ORM. This unlocks multi-instance deployment, proper concurrent writes, full-text search (useful for corpus search), and connection pooling.
Corpus indexing: Store pre-computed TF-IDF vectors in the database as JSONB columns, invalidated when new corpus documents are added. This avoids rebuilding the entire index per submission.
3.6  Security and Authentication
What You Have: The auth system is solid — HMAC-SHA256 hashed API keys, role-based access control with four tiers (super_admin/org_admin/instructor/student), per-org file type allowlists, student quota enforcement, and a master key bootstrap. Rate limiting is in place.
Improvements Needed:
Add HMAC webhook signature: The webhook payload is sent without any signing. A receiver cannot verify the request came from VeriCheck. Add an X-VeriCheck-Signature header using HMAC-SHA256.
CORS is open (*): app.use(cors()) allows any origin. In production, restrict to known frontend origins.
No HTTPS enforcement: There is no redirect from HTTP or HSTS header. Add a middleware to enforce HTTPS.
Per-org rate limiting: Current rate limits are per IP globally. An org with many users could exhaust limits set for another org. Add per-org/per-key rate limiting.
File validation beyond extension: The extension check can be bypassed by renaming files. Add magic-byte validation (file-type npm package) to verify actual MIME type.
3.7  Testing and Validation
Current State: Zero tests. The package.json test script outputs an error and exits. This is the most critical quality gap.
Without tests, it is impossible to know whether algorithm changes improve or regress accuracy, whether API contracts are maintained across refactors, or whether edge cases (empty document, binary file, 50,000-word document, Unicode-only content) are handled correctly.
Minimum Test Suite Needed:
Unit tests for all utility functions: test winnowing similarity on known pairs, test TF-IDF cosine with hand-computed expected values, test obfuscation detection with a string containing Cyrillic homoglyphs, test sentence splitter on edge cases.
Integration tests for the API endpoints: submit a known document, poll until complete, verify the report structure matches the TypeScript interface.
Accuracy benchmarks: a labelled dataset of 50-100 documents with known human/AI labels and known plagiarism percentages. Run the engine and verify precision/recall meet targets.
Regression tests: after any algorithm change, run the benchmark suite and compare scores to a stored baseline.
3.8  LMS Integration
Current State: No Learning Management System integration.
Turnitin's institutional value comes almost entirely from LTI integration — instructors create assignments in Canvas/Moodle/Blackboard, students submit directly from within the LMS, and instructors see reports inline without leaving the LMS.
To reach this level: Implement the IMS Global LTI 1.3 specification. The ltijs npm package handles the OAuth2/OIDC handshake, deep linking, and grade passback. This is a multi-week effort but is required for institutional adoption.

5. Specific Code-Level Issues to Fix
5.1  queue.ts — No retry logic on job failure
The worker calls processOneQueuedJob() every 800ms. If a job fails, it is marked 'failed' with no retry. Add exponential backoff retry (up to 3 attempts) before marking as permanently failed. This prevents transient errors (network timeouts on live corpus fetch) from permanently failing submissions.
5.2  db.ts — Synchronous writes block the event loop
fs.writeFileSync in the persist() function blocks the Node.js event loop on every write. Under concurrent submissions, this creates queuing delays. Replace with fs.writeFile (async) or, better, migrate to SQLite which handles concurrent writes correctly.
5.3  index.ts — CORS is open to all origins
app.use(cors()) allows any origin to make authenticated requests. Change to cors({ origin: process.env.ALLOWED_ORIGINS?.split(',') ?? 'http://localhost:5173' }) and set ALLOWED_ORIGINS in your production environment.
5.4  index.ts — Multer stores orignal filename unsanitised
The uploaded file path uses req.file.originalname directly: `${Date.now()}-${randomUUID()}-${file.originalname}`. A filename containing ../ or special characters could cause path traversal issues. Sanitise with path.basename() and strip non-alphanumeric characters except dots and hyphens.
5.5  ai.ts — COMMON_WORD_PROBS tables are too small
The two probability tables (COMMON_WORD_PROBS_A and _B) each contain 24 words. The cross-perplexity ratio proxy relies on these tables to approximate GPT log-probabilities. With only 24 entries, most tokens receive the fallback eps = 1e-6 probability, which makes the scores noisy and dominated by out-of-vocabulary terms. If not replacing with a real model, expand to at least 10,000 words using a unigram frequency list (e.g., from the Google Books Ngrams corpus).
5.6  plagiarism.ts — O(n×m) sentence comparison is slow at scale
The inner loop in scoreSentenceSimilarity iterates over every corpus sentence for every query sentence. With a corpus of 10,000 sentences and a 200-sentence document, this is 2 million comparisons per submission. Add an inverted index over TF-IDF tokens to filter corpus candidates to the top-100 before running Winnowing and shingle scoring.
5.7  live-web-corpus.ts — No caching of external API results
Every submission fires fresh API requests to OpenAlex, Crossref, and Wikipedia, even for the same query text. Add a simple SHA1-keyed in-memory cache (or SQLite cache) with a 24-hour TTL. This reduces external API dependency, speeds up scanning, and avoids hitting rate limits.
5.8  Client types are duplicated from server types
The TypeScript interfaces in App.tsx (lines 5-100) are manually duplicated from server/src/types.ts. Any server-side type change requires a manual client update. Extract to a shared types package in a monorepo root (e.g., packages/types/) or generate client types from the API schema using openapi-typescript.
6. What VeriCheck AI Already Does Well
It is important to recognise the significant strengths already present. These should be preserved and built upon:
Multi-pass plagiarism engine: Winnowing + TF-IDF + shingles is a mature, well-referenced approach. The corpus indexing strategy (pre-computing vectors per scan) is correct.
Obfuscation detection and normalisation: The homoglyph map, zero-width character stripping, and full-width ASCII normalisation are all features that many plagiarism checkers lack.
DOCX forensic metadata: Extracting author, revision count, edit time, tracked changes, font families, and generating suspicion flags is genuinely impressive and differentiated.
Role hierarchy and org config: The four-role system (super_admin/org_admin/instructor/student) with per-org configuration is production-grade institutional architecture.
Async job queue with webhooks and email: Using a poll-based queue with webhook callbacks and email notifications is the correct pattern for long-running analysis jobs.
Scan versioning: The parent_job_id / scan_version tracking for resubmissions is a differentiating feature that Turnitin charges extra for.
